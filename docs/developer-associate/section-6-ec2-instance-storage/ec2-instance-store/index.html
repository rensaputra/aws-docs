EC2 Instance Store

EC2 Instance Store is the ultimate "speed demon" of the AWS storage world. If EBS is like plugging in a network-attached USB stick, an Instance Store is like soldering a blazing-fast NVMe SSD directly onto the physical motherboard of the server host.

Key Takeaways​

The Core Performance Advantage​

  • Physical Attachment: Unlike EBS (which sends data back and forth over a network wire), an instance store is physically connected to the underlying hardware server hosting your virtual machine.
  • Raw Power It delivers insanely high I/O performance. Stephane points out that an Instance Store can hit millions of IOPS (e.g., 3.3 million reads), while a standard gp2 EBS volume maxes out around 32,000 IOPS.

The Catch: Ephemeral Storage​

  • The Wipe Rule: This storage is strictly ephemeral (temporary).
  • The Danger Zone: If you STOP or TERMINATE the instance, or if the underlying physical hardware suffers a hardware failure, the data is gone forever. You cannot get it back. -Survival Mode: The data only survives a standard OS reboot.

Ideal Use Cases​

Because it's fast but fragile, you only use it for data that can be lost without causing a production disaster.

  • High-performance caches and buffers.
  • Scratch space or temporary file processing directories.
  • Distributed file systems or databases that handle replication at the software level (where losing one node doesn't kill the cluster).

Exam Tips​

The "Stop" Limitation: For EC2 instances backed by only by an Instance Store, **the "Stop" action is completely disabled in the AWS Console/CLI". You can only "Reboot" or "Terminate" them. Why? Because stopping them means moving them to a different physical host, which would wipe the drive anyway.

Keyword Association: On the exam, if you see phrases like maximum I/O performance, Lowest Latency, local storage, or "non-persistent data", your mind should immediately skip EBS and choose Instance Store.

The Backup Responsibility: If a question asks who is responsible for backing up instance store data, it 100% on the developer. AWS will not snapshot or replicate this data for you. You have to write scripts to copy it to S3 or EBS manually.
